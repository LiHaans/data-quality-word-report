# 天文数据实时全局去重技术实现方案

## 1. 文档目标

本文档面向如下业务场景，给出一套**可实施、可扩展、兼顾实时性与全局去重能力**的技术方案。

### 1.1 业务背景

当前数据链路为：

```text
接口推送 -> Kafka 原始 Topic -> Flink SQL / Flink 作业处理 -> Kafka 结果 Topic -> Elasticsearch 8
```

数据特点：

- 数据类型：天文数据
- 日新增量：约 **1000 万**
- 历史累计量：约 **50 多亿**
- 数据持续增长
- 数据时间跨度很大：
  - 可能是今天
  - 可能是昨天
  - 可能是去年
  - 也可能是很多年前
- 需要基于业务字段生成唯一值进行**全局去重**
- 同时要求数据处理具备较好的**实时性**，不希望最终结果延迟过高

### 1.2 去重需求

去重规则基于如下字段组合生成唯一键：

- 标题
- 发布时间
- 站点名称
- 其他必要辅助字段（如来源标识、站点编码等）

要求：

- 对全量历史数据范围进行唯一性判断
- 新数据流入时实时判重
- 支持历史补数、乱序补数、跨年份补数
- 支持持续新增

---

## 2. 结论先行

### 2.1 最终推荐方案

推荐采用如下架构：

```text
接口推送
  -> Kafka 原始 Topic
  -> Flink SQL：字段清洗、标准化、生成 dedup_key
  -> Flink DataStream：短期热去重（RocksDB State + TTL）
  -> Flink Async I/O：异步查询/写入全局去重索引（HBase）
  -> Kafka 去重结果 Topic
  -> Elasticsearch 8（_id = dedup_key）
```

### 2.2 方案核心思想

本方案采用**三层去重机制**：

1. **第一层：Flink 本地热去重**
   - 用于拦截短时间内重复数据
   - 降低全局索引查询压力
   - 实现毫秒级过滤

2. **第二层：HBase 全局去重索引**
   - 用于做真正的永久全局判重
   - 支持几十亿级唯一键长期存储
   - 支持跨年份历史补数去重

3. **第三层：ES 幂等写入兜底**
   - 使用 `dedup_key` 作为文档 `_id`
   - 防止极端并发或重试导致的重复写入

### 2.3 为什么不建议纯 Flink SQL 永久去重

原因如下：

- 50 多亿历史唯一键无法长期优雅存储在 Flink 状态中
- 状态会无限膨胀
- checkpoint 成本会不断升高
- 恢复时间会越来越长
- 对“跨年补数”的全局唯一性支持不经济

因此：

> **Flink SQL 负责标准化与流式处理，永久全局唯一索引必须下沉到外部存储。**

## 4. 去重键设计

## 4.1 核心原则

去重键不能直接使用原始字段拼接，必须先进行标准化。

否则以下情况会导致同一条数据生成不同 key：

- 标题前后空格不同
- 中英文符号不同
- 站点名称写法不同
- 发布时间时区不同
- 时间精度不同

---

## 4.2 字段标准化规则

### 4.2.1 标题标准化

建议规则：

1. 去除首尾空格
2. 连续空白折叠为单空格
3. 全角转半角
4. 统一常见标点
5. 可选：统一大小写（仅适用于英文场景）
6. 去除明显无意义前缀或后缀（若业务确认安全）

示例：

```text
原始标题：  【NASA】发现新彗星！  
标准标题：NASA 发现新彗星!
```

---

### 4.2.2 站点名称标准化

建议不要直接使用原始站点名称，而是维护一套映射表：

```text
NASA官网 -> nasa
Nasa -> nasa
美国国家航空航天局 -> nasa
```

即：

- `site_name`：原始站点名称
- `site_code`：标准站点编码

去重时优先使用 `site_code`。

---

### 4.2.3 发布时间标准化

发布时间必须统一：

- 时区统一为 UTC 或东八区
- 精度统一到秒（推荐）
- 输出统一时间戳

例如：

```text
2026-04-07 00:01:02+08:00 -> 1775491262
```

若来源时间精度不可靠，可按业务实际选择：

- 精确到秒
- 精确到分钟
- 精确到日期

通常建议：

> **优先精确到秒；若源站发布时间不稳定，再退化到分钟。**

---

## 4.3 去重键生成公式

建议使用：

```text
dedup_source = normalized_title + "|" + publish_time_epoch + "|" + site_code
dedup_key = sha256(dedup_source)
```

### 4.3.1 为什么推荐 SHA-256

优点：

- 碰撞概率极低
- 输出固定长度
- 适合做 HBase rowkey 和 ES `_id`

如果追求更短，也可以考虑：

- MD5
- MurmurHash64

但从长期安全性和碰撞控制来看，推荐优先使用 SHA-256。

---

## 4.4 推荐保留的辅助字段

除了 `dedup_key`，建议同时保留：

- `dedup_source`（可选）
- `normalized_title`
- `site_code`
- `publish_time_epoch`
- `ingest_time`
- `raw_id` / 来源原始主键

这些字段对后续问题排查非常有帮助。

---

## 5. 主题设计（Kafka Topic）

建议至少拆分如下 Topic：

### 5.1 原始数据 Topic

- `astro_raw`

作用：
- 保存接口原始推送数据
- 作为原始事实输入

---

### 5.2 标准化数据 Topic（可选）

- `astro_normalized`

作用：
- 保存已清洗、已标准化、已生成 `dedup_key` 的中间结果
- 方便问题排查、回放与下游复用

---

### 5.3 去重结果 Topic

- `astro_clean`

作用：
- 保存通过全局去重后的数据
- 提供给下游 ES 或其他系统消费

---

### 5.4 重复数据 Topic（建议保留）

- `astro_duplicate`

作用：
- 保存被识别为重复的数据
- 便于审计、回溯和规则调优

---

### 5.5 全局索引失败重试 Topic（建议保留）

- `astro_dedup_retry`

作用：
- 当 HBase 暂时不可用、超时、写入失败时，进入重试或补偿链路

## 6. HBase 全局去重索引设计

## 6.1 为什么选择 HBase

HBase 适合当前场景的原因：

- 支持海量 Key 持久化存储
- 随机读写能力较好
- 对几十亿级索引表更现实
- 能配合原子写入机制实现全局唯一判定
- 与 Flink / Hadoop 体系兼容度较高

---

## 6.2 表设计

建议建立 HBase 表：

- 表名：`astro_global_dedup_index`

列族：

- `f`

### 推荐字段

| 列 | 含义 |
|----|------|
| rowkey | dedup_key |
| f:first_seen_time | 首次入库时间 |
| f:publish_time | 发布时间 |
| f:site_code | 标准站点编码 |
| f:es_id | ES 文档 ID（可与 dedup_key 相同） |
| f:raw_ref | 原始消息引用（可选） |

---

## 6.3 RowKey 设计建议

直接用 `dedup_key` 可能会导致局部热点，建议加散列前缀，例如：

```text
rowkey = prefix + '_' + dedup_key
prefix = dedup_key 前 2~4 位
```

示例：

```text
a3_9b39b3f6f7c1...
```

这样可以将写入打散到不同 Region，降低热点风险。

---

## 6.4 HBase 数据规模说明

当前累计 50 多亿，且持续增长，因此去重索引本身就是长期核心数据资产。

建议：

- HBase 集群单独规划
- 保持足够 Region 数量
- 做容量与压缩规划
- 定期评估热点 key 分布

---

## 7. 实时去重策略设计

## 7.1 三层去重策略

### 第一层：Flink 热去重

目标：
- 拦截近实时重复数据
- 降低全局索引访问量

实现：
- 按 `dedup_key` 做 keyBy
- 使用 RocksDB State 保存短期 dedup key
- 为状态配置 TTL

建议 TTL：

- 3 天（推荐起步）
- 或 7 天（如果短期重复很多）

### 为什么不能把 TTL 设成永久

因为：
- 状态会无限膨胀
- checkpoint 成本不可控
- 不适合 50 亿级永久唯一集合

所以：

> 热去重层只负责拦截近实时重复，不负责永久全局唯一。

---

### 第二层：HBase 全局判重

目标：
- 支持任意时间跨度的永久全局去重
- 支持历史补数与跨年补数

逻辑：

1. 对未命中热去重的数据，查 HBase
2. 如果 dedup_key 已存在，则判定为重复
3. 如果 dedup_key 不存在，则原子写入索引并放行

---

### 第三层：ES 幂等写入

目标：
- 防止极端情况下并发穿透
- 防止重复写入 ES

实现：

- ES 文档 `_id = dedup_key`

即使上游偶发重复穿透，ES 中最终也只保留一个文档。

---

## 7.2 为什么不能只用窗口去重

窗口去重适用于：

- 最近 5 分钟
- 最近 1 小时
- 最近 1 天

而你的场景是：

- 可能今天补去年数据
- 可能今天补很多年前数据

所以窗口去重无法满足全局唯一要求。

---

## 8. 实时性保障设计

## 8.1 延迟目标

本方案目标延迟控制在：

- 正常情况下：**1~5 秒**
- 峰值情况下：**不超过十几秒**

不建议方案演化为分钟级延迟。

---

## 8.2 为什么用 Async I/O

如果对每条数据都同步查 HBase，会出现：

- 单条阻塞
- 吞吐下降
- 延迟升高

因此建议：

> **使用 Flink Async I/O 异步访问 HBase。**

优点：

- 并发访问外部存储
- 提高吞吐
- 降低处理链路阻塞
- 更适合每分钟持续有新数据的流式场景

---

## 8.3 建议的实时处理链路

```text
Kafka -> Flink SQL 清洗 -> 热去重 -> Async HBase 全局判重 -> Kafka clean -> ES
```

各阶段延迟预期：

- SQL 清洗：毫秒级~亚秒级
- 热去重：毫秒级
- Async HBase：几十毫秒~几百毫秒（视集群而定）
- Kafka + ES：秒级

---

## 9. 并发一致性设计

## 9.1 问题说明

如果流程是：

1. 查 HBase，不存在
2. 再写 HBase

在高并发下可能出现：

- A 查不到
- B 查不到
- A 写入
- B 写入

导致重复穿透。

---

## 9.2 解决方案

### 优先方案：原子写入

推荐使用 HBase 的原子语义，如：

- `checkAndMutate`
- 或基于“列不存在才写入”的原子操作

逻辑：

- 如果 rowkey 不存在，则写入成功，并判定为新数据
- 如果 rowkey 已存在，则写入失败，并判定为重复数据

---

### 兜底方案：ES `_id = dedup_key`

即使极端情况下 HBase 层偶发穿透，ES 仍可通过 `_id` 保证最终文档唯一。

---

## 10. Elasticsearch 8 写入设计

## 10.1 文档 ID 设计

必须使用：

```text
_id = dedup_key
```

优点：

- 重试写入不产生重复文档
- 重复消息覆盖同一文档
- 保证最终落 ES 唯一

---

## 10.2 建议保留字段

| 字段 | 说明 |
|------|------|
| dedup_key | 去重唯一键 |
| title | 原始标题 |
| normalized_title | 标准化标题 |
| publish_time | 标准发布时间 |
| site_name | 原始站点名 |
| site_code | 标准站点编码 |
| ingest_time | 入流时间 |
| raw_payload | 原始数据（可选） |

---

## 10.3 ES 写入建议

建议：

- 使用 bulk 写入
- 合理设置批大小
- 控制 refresh interval
- 不要每条单独强制刷新

## 11. Flink 实现方式建议

## 11.1 为什么不建议纯 SQL 完成全链路

纯 Flink SQL 比较适合：

- 数据清洗
- 字段加工
- 规则标准化
- 短期窗口计算

但对以下能力支持不够优雅：

- 永久全局去重索引
- Async 外部判重
- 原子写索引
- 失败补偿与重试控制

因此建议采用：

### 混合模式

- **Flink SQL**：负责字段清洗与 `dedup_key` 生成
- **Flink DataStream**：负责热去重、异步全局判重、侧输出、失败重试

---

## 11.2 推荐拆分方式

### 作业 1：Flink SQL 标准化作业

输入：
- `astro_raw`

输出：
- `astro_normalized`

职责：
- 标题标准化
- 站点标准化
- 发布时间标准化
- 生成 `dedup_key`

---

### 作业 2：Flink 去重主作业

输入：
- `astro_normalized`

职责：
- 热去重
- Async HBase 全局判重
- 新数据输出到 `astro_clean`
- 重复数据输出到 `astro_duplicate`
- 异常输出到 `astro_dedup_retry`

---

## 12. Flink SQL 示例（标准化阶段）

> 以下为示意 SQL，具体函数需根据实际 UDF / connector 调整。

```sql
CREATE TABLE astro_raw (
    title STRING,
    publish_time STRING,
    site_name STRING,
    raw_payload STRING,
    ingest_time TIMESTAMP(3),
    proctime AS PROCTIME()
) WITH (...);

CREATE TABLE astro_normalized (
    dedup_key STRING,
    title STRING,
    normalized_title STRING,
    publish_time STRING,
    publish_time_epoch BIGINT,
    site_name STRING,
    site_code STRING,
    raw_payload STRING,
    ingest_time TIMESTAMP(3)
) WITH (...);

INSERT INTO astro_normalized
SELECT
    SHA2(CONCAT(
        normalize_title(title), '|',
        CAST(to_unix_ts(normalize_publish_time(publish_time)) AS STRING), '|',
        normalize_site(site_name)
    ), 256) AS dedup_key,
    title,
    normalize_title(title) AS normalized_title,
    normalize_publish_time(publish_time) AS publish_time,
    to_unix_ts(normalize_publish_time(publish_time)) AS publish_time_epoch,
    site_name,
    normalize_site(site_name) AS site_code,
    raw_payload,
    ingest_time
FROM astro_raw;
```

说明：

- `normalize_title`：自定义 UDF
- `normalize_site`：自定义 UDF
- `normalize_publish_time`：自定义 UDF
- `to_unix_ts`：转换为时间戳

## 13. Flink DataStream 伪代码（主去重作业）

```java
stream
  .keyBy(record -> record.getDedupKey())
  .process(new HotDedupProcessFunction(ttlDays = 3))
  .filter(record -> !record.isDuplicateInHotState())
  .asyncWait(
      new GlobalDedupAsyncFunction(hbaseClient),
      timeout = 500,
      capacity = 1000
  )
  .process(new RouteResultFunction())
```

### 13.1 热去重逻辑

```java
if (state.contains(dedupKey)) {
    output duplicate side output;
} else {
    state.put(dedupKey, true);
    forward to global dedup;
}
```

### 13.2 全局去重逻辑

```java
if (hbase.checkAndPut(dedupKey)) {
    // 写入成功，说明是新数据
    emit clean stream;
} else {
    // 已存在，说明重复
    emit duplicate side output;
}
```

## 14. 失败补偿设计

## 14.1 为什么需要补偿

如果 HBase 短暂不可用、网络超时、集群抖动，不能直接让整条链路完全失败。

建议保留失败补偿通道。

---

## 14.2 建议做法

### 失败侧输出 Topic

- `astro_dedup_retry`

当出现以下情况时写入该 Topic：

- HBase 超时
- 异步查询失败
- 原子写入失败
- 短时网络抖动

### 重试策略

- 限制最大重试次数
- 指数退避
- 超限后打到人工处理 Topic 或死信队列

---

## 15. 容量与性能建议

## 15.1 Flink 层建议

- StateBackend：RocksDB
- 开启增量 checkpoint
- 热去重 TTL：建议 3 天起步
- Async I/O：控制并发度与 timeout
- checkpoint 间隔按集群能力配置

---

## 15.2 HBase 层建议

- 独立集群承载去重索引
- RowKey 做前缀打散
- 预分区 / 预建 Region
- 评估压缩与 BloomFilter
- 监控热点 region

---

## 15.3 Kafka 层建议

- 原始 Topic 与 clean Topic 分离
- duplicate Topic 独立保留
- retry Topic 独立保留
- partition 数量按吞吐与并行度匹配

---

## 15.4 ES 层建议

- `_id = dedup_key`
- bulk 写入
- 控制 refresh interval
- 索引模板提前规划字段类型

---

## 16. 延迟与吞吐预估

在合理配置下，本方案可实现：

### 正常延迟

- 热去重阶段：毫秒级
- 异步全局判重：几十毫秒到几百毫秒
- 整体结果入 Kafka/ES：通常 1~5 秒

### 峰值情况下

- 可控制在十几秒内
- 不建议退化成分钟级

---

## 17. 实施步骤建议

## 第一阶段：基础能力上线

目标：先跑通主链路

步骤：

1. 完成字段标准化规则定义
2. 完成 `dedup_key` 生成规则
3. 建 Kafka 原始 / clean / duplicate Topic
4. 建 HBase 去重索引表
5. Flink SQL 实现标准化作业
6. Flink DataStream 实现热去重 + HBase 判重
7. ES 写入 `_id = dedup_key`

---

## 第二阶段：稳定性增强

1. 增加 retry Topic
2. 增加失败补偿机制
3. 增加 duplicate 数据审计
4. 监控 HBase 延迟、热点、失败率
5. 监控 Flink async backlog

---

## 第三阶段：性能优化

1. 调整热去重 TTL
2. 根据重复特征增加 Bloom Filter 层
3. 做 HBase RowKey 打散优化
4. 优化 ES bulk 写入策略
5. 优化 Kafka partition 与 Flink 并行度

## 18. 不推荐方案说明

以下方案不推荐作为主方案：

### 18.1 纯窗口去重
原因：
- 无法覆盖跨年补数
- 不满足全局唯一

### 18.2 纯 Flink 永久 State 去重
原因：
- 状态无限膨胀
- checkpoint 和恢复不可控

### 18.3 纯 Redis 永久去重
原因：
- 50 亿级长期索引成本过高
- 内存型系统不经济

### 18.4 纯 ES 查询判重
原因：
- 检索成本高
- 不适合作为高频全局去重主索引
- 容易影响检索集群稳定性

## 19. 最终方案总结

本方案的最终推荐为：

### 推荐技术组合

- Kafka：消息入口与结果输出
- Flink SQL：字段清洗与标准化
- Flink DataStream：热去重 + Async 全局判重
- HBase：永久全局去重索引
- Elasticsearch 8：最终结果检索存储

### 核心设计原则

1. **先标准化，再生成唯一键**
2. **先热去重，再全局判重**
3. **全局索引下沉到外部存储**
4. **ES 使用 `_id = dedup_key` 做最终幂等**
5. **通过 Async I/O 保证实时性**

### 最终能力

该方案能够支持：

- 日增 1000 万级实时处理
- 50 亿级历史数据全局去重
- 跨年、跨历史时期补数
- 低延迟输出
- 长期扩展与持续增长

## 20. 一句话结论

对于“天文数据、日增千万、累计数十亿、时间跨度无限、要求全局去重且实时性较高”的场景，**最可实施的方案是：Flink SQL 负责标准化，Flink DataStream + HBase 负责全局判重，ES 以 dedup_key 为 `_id` 做最终幂等落库。**

## 21. 深化版实施方案（Redis 第一层 + HBase 第二层）

本章节在前文基础上，将方案进一步细化为真正可落地的工程实施版本。

本版方案采用：

1. **第一层：Redis 热去重层**
2. **第二层：HBase 永久全局去重层**
3. **第三层：ES 幂等写入层**

也就是：

```text
Kafka -> Flink SQL 标准化 -> Redis 热去重 -> HBase 全局去重 -> Kafka clean -> ES
```

### 为什么第一层改用 Redis

你明确提出希望第一层使用 Redis 过滤。这个设计是合理的，原因如下：

- Redis 适合做**近实时热数据快速判重**
- 延迟低，适合毫秒级响应
- 对日增 1000 万、分钟级持续流入的短周期重复过滤很有效
- 可减少 HBase 的读写压力

但必须强调：

> **Redis 只适合做第一层热去重，不能承担 50 多亿永久全局索引主库。**

## 22. 最终推荐链路（实施版）

```text
接口推送
  -> Kafka: astro_raw
  -> Flink SQL 标准化
  -> Flink 主作业
      1. 生成 dedup_key
      2. Redis SETNX/EX 做热去重
      3. 未命中热重复 -> Async HBase check-and-put 全局判重
      4. 新数据 -> Kafka astro_clean
      5. 重复数据 -> Kafka astro_duplicate
      6. 失败数据 -> Kafka astro_dedup_retry
  -> Elasticsearch 8 (_id = dedup_key)
```

## 23. Redis 第一层热去重设计

## 23.1 第一层目标

Redis 第一层解决的是：

- 短时间内重复推送
- Kafka at-least-once 导致的短周期重复消费
- 接口重试导致的短周期重复
- 同站点短时间重复发布

### 第一层不解决什么

Redis 第一层**不解决**：

- 去年的数据今天补进来
- 历史任意时期全局唯一
- 永久索引唯一性

这些由 HBase 第二层承担。

---

## 23.2 Redis 存储模型

建议使用如下键设计：

```text
key   = astro:dedup:hot:{dedup_key}
value = 1
TTL   = 3天 或 7天
```

推荐命令：

```text
SET key 1 NX EX <ttl_seconds>
```

含义：

- 如果 key 不存在，则写入成功，说明是热层首次出现
- 如果 key 已存在，则说明在热去重范围内重复，直接判重

---

## 23.3 Redis TTL 建议

建议值：

- 起步：`3 天`
- 如果短期重复特别多：可提高到 `7 天`
- 如果 Redis 压力较大：可退到 `1 天`

推荐默认：

```text
TTL = 3 天 = 259200 秒
```

原因：

- 大多数短周期重复会发生在分钟级、小时级、天级
- 3 天通常是一个比较均衡的折中
- 既能挡住大部分短期重复，又不会让 Redis 成本爆炸

---

## 23.4 Redis 容量估算（第一层）

### 已知数据量

- 日新增：1000 万
- 假设 3 天 TTL
- 理论活跃 key 规模：约 3000 万

### 粗略内存估算

若每个 dedup key 按以下估算：

- key 长度：约 80~110 字节（含前缀）
- value：极小
- Redis 对象结构开销：较大

实际每条可能在 Redis 中占用：

- **120B ~ 250B**（粗略工程估值）

则：

#### 按 150B 估算

```text
3000 万 * 150B ≈ 4.5 GB
```

#### 按 220B 估算

```text
3000 万 * 220B ≈ 6.6 GB
```

### Redis 资源建议

建议至少：

- **Redis Cluster 3 主 3 从** 或 **3 主（如能接受降冗余）**
- 单主节点内存建议：`8GB ~ 16GB`
- 总内存建议预留：`24GB+`

如果只做第一层热去重且 TTL 控制在 3 天，这个规模是可控的。

---

## 23.5 Redis 第一层处理逻辑

伪逻辑如下：

```text
1. 计算 dedup_key
2. redis SET key 1 NX EX 259200
3. 如果返回成功：说明未命中热重复，进入 HBase 全局去重
4. 如果返回失败：说明命中热重复，直接丢弃或输出 duplicate
```

### 优点

- 非常快
- 毫秒级
- 大幅降低 HBase 压力

---

## 24. HBase 第二层永久全局去重设计

## 24.1 第二层目标

HBase 第二层用于承担：

- 永久全局唯一索引
- 历史任意时期补数判重
- 跨年、跨多年历史重复判断

也就是说：

> **Redis 负责热，HBase 负责全局。**

---

## 24.2 HBase 表结构（实施版）

### 表名

```text
astro_global_dedup_index
```

### 列族

```text
f
```

### RowKey

推荐：

```text
rowkey = prefix + '_' + dedup_key
prefix = dedup_key 前 2 位或前 4 位
```

例如：

```text
9f_9f6e2cb8e0c7c5e0b1...
```

### 字段设计

| 列 | 示例 | 说明 |
|----|------|------|
| rowkey | 9f_xxx | 主键 |
| f:first_seen_time | 1775491262 | 首次入库时间 |
| f:publish_time | 1775400000 | 标准发布时间 |
| f:site_code | nasa | 标准站点编码 |
| f:doc_type | astro | 业务类型 |
| f:es_id | dedup_key | ES 文档 ID |
| f:source_ref | topic-partition-offset | 原始引用 |

实际上最简模式中，只要存在 rowkey 就够判重。但建议保留基础审计字段。

---

## 24.3 HBase 原子去重策略

### 推荐方式

使用 HBase 的原子操作语义：

- `checkAndMutate`
- `checkAndPut`
- 或“列不存在才写入”语义

逻辑：

```text
如果 row 不存在 -> 写入成功 -> 新数据
如果 row 已存在 -> 写入失败 -> 重复数据
```

这样可以解决并发穿透问题。

---

## 24.4 HBase 资源估算

这是工程实施必须考虑的。

### 已知规模

- 历史累计：50 亿+
- 日增：1000 万+
- 持续增长

### 单条索引大小粗估

假设：

- rowkey：约 70~90B
- 列族/列名/时间戳/版本等开销
- HFile / MemStore / BlockIndex / BloomFilter 开销

工程上通常不要按“纯 key 字节数”估算，而要按**100B~300B/条**的实际存储开销估算。

#### 保守估算 150B/条

```text
50 亿 * 150B ≈ 750 GB
```

#### 保守估算 250B/条

```text
50 亿 * 250B ≈ 1.25 TB
```

### 实际建议

建议按 **1.5TB ~ 3TB 有效存储规模** 规划起步容量，原因：

- 压缩前后差异
- HBase 索引开销
- 未来增长空间
- 副本与预留空间

## 24.5 HBase 集群节点建议

### 起步建议（生产可实施版）

#### RegionServer 节点

建议：

- **6~8 台 RegionServer** 起步

每台建议配置：

- CPU：16C ~ 32C
- 内存：64GB ~ 128GB
- 磁盘：2TB+ SSD / NVMe
- JVM Heap：16GB ~ 24GB（不要太大）
- 其余给 BlockCache / OS Cache / MemStore

#### Master 节点

建议：

- 2 台 HMaster（主备）
- 配置：8C / 16C，32GB 内存即可

#### ZooKeeper

建议：

- 3 台独立 ZooKeeper 节点

## 24.6 HBase 预分区建议

不要等表自动分裂到很大后再处理。

建议按 rowkey 前缀预建 region，例如：

- 如果 prefix 用前 2 位 hex
- 理论可拆成 256 个前缀

可以起步预分 64~128 个 region。

例如：

```text
00_, 04_, 08_, 0c_, ...
```

这样写入更均匀。

## 24.7 HBase 压缩与优化建议

建议：

- 压缩：Snappy
- BloomFilter：ROW
- BlockCache：开启
- 定期 compact
- 合理控制 MemStore flush


## 27.1 标题标准化 UDF（Java 示例）

```java
package com.example.udf;

import org.apache.flink.table.functions.ScalarFunction;

public class NormalizeTitleUdf extends ScalarFunction {

    public String eval(String title) {
        if (title == null) {
            return null;
        }
        String s = title.trim();
        s = toHalfWidth(s);
        s = s.replaceAll("\\s+", " ");
        s = s.replace('【', '[').replace('】', ']');
        s = s.replace('（', '(').replace('）', ')');
        s = s.replace('，', ',').replace('。', '.').replace('：', ':').replace('！', '!').replace('？', '?');
        return s;
    }

    private String toHalfWidth(String input) {
        char[] chars = input.toCharArray();
        for (int i = 0; i < chars.length; i++) {
            if (chars[i] == 12288) {
                chars[i] = ' ';
            } else if (chars[i] >= 65281 && chars[i] <= 65374) {
                chars[i] = (char) (chars[i] - 65248);
            }
        }
        return new String(chars);
    }
}
```

## 27.4 dedup_key 生成 UDF（Java 示例）

```java
package com.example.udf;

import org.apache.flink.table.functions.ScalarFunction;

import java.nio.charset.StandardCharsets;
import java.security.MessageDigest;

public class DedupKeyUdf extends ScalarFunction {

    public String eval(String normalizedTitle, Long publishTimeEpoch, String siteCode) {
        if (normalizedTitle == null || publishTimeEpoch == null || siteCode == null) {
            return null;
        }
        String source = normalizedTitle + "|" + publishTimeEpoch + "|" + siteCode;
        return sha256(source);
    }

    private String sha256(String s) {
        try {
            MessageDigest digest = MessageDigest.getInstance("SHA-256");
            byte[] hash = digest.digest(s.getBytes(StandardCharsets.UTF_8));
            StringBuilder sb = new StringBuilder();
            for (byte b : hash) {
                sb.append(String.format("%02x", b));
            }
            return sb.toString();
        } catch (Exception e) {
            throw new RuntimeException(e);
        }
    }
}
```

## 29.1 Redis 热去重示意代码

```java
public class RedisHotDedupProcessFunction extends KeyedProcessFunction<String, AstroRecord, AstroRecord> {

    private transient JedisPool jedisPool;
    private transient OutputTag<AstroRecord> duplicateTag;

    @Override
    public void open(Configuration parameters) {
        this.jedisPool = new JedisPool("redis-host", 6379);
        this.duplicateTag = new OutputTag<AstroRecord>("duplicate"){};
    }

    @Override
    public void processElement(AstroRecord value, Context ctx, Collector<AstroRecord> out) throws Exception {
        String key = "astro:dedup:hot:" + value.getDedupKey();
        try (Jedis jedis = jedisPool.getResource()) {
            String result = jedis.set(key, "1", SetParams.setParams().nx().ex(259200));
            if ("OK".equals(result)) {
                out.collect(value);
            } else {
                ctx.output(duplicateTag, value);
            }
        }
    }
}
```

## 29.2 HBase 全局去重 Async 示例思路

```java
public class HBaseGlobalDedupAsyncFunction extends RichAsyncFunction<AstroRecord, AstroRecord> {

    private transient AsyncConnection connection;
    private transient Table table;

    @Override
    public void open(Configuration parameters) throws Exception {
        org.apache.hadoop.conf.Configuration conf = HBaseConfiguration.create();
        connection = ConnectionFactory.createAsyncConnection(conf).get();
        table = connection.getTable(TableName.valueOf("astro_global_dedup_index"));
    }

    @Override
    public void asyncInvoke(AstroRecord input, ResultFuture<AstroRecord> resultFuture) {
        String rowKey = input.getDedupKey().substring(0, 2) + "_" + input.getDedupKey();

        Put put = new Put(Bytes.toBytes(rowKey));
        put.addColumn(Bytes.toBytes("f"), Bytes.toBytes("first_seen_time"), Bytes.toBytes(String.valueOf(System.currentTimeMillis())));
        put.addColumn(Bytes.toBytes("f"), Bytes.toBytes("publish_time"), Bytes.toBytes(String.valueOf(input.getPublishTimeEpoch())));
        put.addColumn(Bytes.toBytes("f"), Bytes.toBytes("site_code"), Bytes.toBytes(input.getSiteCode()));

        // 生产中这里应使用 checkAndMutate / checkAndPut 原子语义
        table.checkAndMutate(Bytes.toBytes(rowKey), Bytes.toBytes("f"))
            .qualifier(Bytes.toBytes("first_seen_time"))
            .ifNotExists()
            .thenPut(put)
            .whenComplete((ok, ex) -> {
                if (ex != null) {
                    resultFuture.completeExceptionally(ex);
                } else if (Boolean.TRUE.equals(ok)) {
                    resultFuture.complete(java.util.Collections.singleton(input));
                } else {
                    resultFuture.complete(java.util.Collections.emptyList());
                }
            });
    }
}
```

> 注意：实际版本要按你们使用的 HBase Client 版本调整 API。

## 30.2 Redis 不可用时策略

### 策略 A：直接绕过 Redis，进入 HBase
优点：
- 保证主链路不停
缺点：
- HBase 压力瞬间增大

### 策略 B：写 retry topic
优点：
- 稳妥
缺点：
- 增加延迟

### 推荐

生产上建议：

- Redis 短时不可用时，**降级直打 HBase**
- 若 HBase 也异常，再写 retry topic
